# Agent编排设计：主控 Agent + 分类处理模块

本文档用于定义智能物业报告项目第一阶段推荐采用的 Agent 编排方式，目标是在控制复杂度的前提下，让系统能够稳定处理不同目录、不同分类、不同类型的原始资料。

---

## 1. 文档目的

当前原始资料存在以下特点：

- 分散在多个目录
- 按不同类型分类
- 更新频率不同
- 文件格式不同
- 需要生成统一月报

因此，系统不能只做一个简单的“读取一个目录然后输出一篇文章”的流程，而应有明确的编排逻辑。

但第一阶段也不适合一开始就上很多彼此独立、复杂协作的多智能体系统。

所以推荐采用：

**一个主控 Agent + 多个分类处理模块**

这是一种“逻辑分工、多步协作、统一编排”的方式。

---

## 2. 为什么第一阶段不建议直接做多个独立 Agent

虽然从概念上可以拆成多个 Agent，例如：

- 财务 Agent
- 合规 Agent
- 运维 Agent
- 报告撰写 Agent

但第一阶段直接做成多 Agent 系统，会有以下问题：

- 调试复杂
- 错误来源不容易定位
- 上下游依赖难管理
- 成本更高
- 试点失败风险更大

对于第一阶段，更重要的是：

- 跑通流程
- 控制范围
- 让业务团队看见价值

因此，不建议一开始就采用复杂的多 Agent 架构。

---

## 3. 第一阶段推荐架构

建议第一阶段采用：

**主控 Agent + 分类处理模块 + 报告生成模块**

其中：

- 主控 Agent 负责总流程编排
- 分类处理模块负责处理不同资料类型或业务领域
- 报告生成模块负责最终成文

这里的“分类处理模块”在第一阶段可以是：

- Agent 子任务
- 内部处理器
- 工作流步骤
- 脚本模块

并不一定要全部实现为真正独立运行的智能体。

---

## 4. 主控 Agent 的职责

主控 Agent 是整个流程的“总调度者”。

### 4.1 主控 Agent 主要负责

- 接收用户输入的物业名称和报告月份
- 判断需要扫描哪些目录
- 区分基础资料与月度资料
- 判断缺少哪些关键输入
- 调用各分类处理模块
- 汇总结构化结果
- 调用报告生成模块
- 输出报告初稿和缺失项清单

### 4.2 主控 Agent 不直接承担所有细节抽取

主控 Agent 应尽量避免直接自己处理所有内容，而是把任务分发给更专门的模块。

这样做的好处：

- 逻辑清晰
- 易于调试
- 便于后续替换和扩展

---

## 5. 推荐的分类处理模块

第一阶段建议按业务类别拆分，而不是按技术格式过度拆分。

推荐至少拆成以下 4 个处理模块。

### 5.1 财务与欠费处理模块

负责读取：

- 财务 Excel
- 欠费清单
- 回款记录
- 水电统计表

输出：

- 财务字段
- 欠费字段
- 公用事业字段

适合提取的核心内容：

- 管理基金收入/支出
- 偿债基金收入/支出
- 欠费总额
- 重点欠费单位
- 水电用量与费用

### 5.2 合同与合规处理模块

负责读取：

- 项目主数据
- 合同台账
- 执照台账
- 保险资料

输出：

- 项目概况字段
- 合同到期事项
- 执照到期事项
- 合规风险字段

适合提取的核心内容：

- 物业基础信息
- 即将到期合同
- 即将到期执照
- 保险到期提醒

### 5.3 运营与现场处理模块

负责读取：

- 工单
- 维修记录
- 安保月报
- 图片
- 巡检记录

输出：

- 维修字段
- 安保字段
- 重大事件字段

适合提取的核心内容：

- 新增/关闭工单
- 重大故障
- 安保事件
- 访客与违停统计
- 图片中识别出的异常事项

### 5.4 治理与待决事项处理模块

负责读取：

- 会议议程
- 会议纪要
- 邮件导出
- 聊天记录导出

输出：

- 待委员会决策事项
- 住户申诉事项
- matters arising
- 管理建议

适合提取的核心内容：

- 需拍板事项
- 投诉与申诉
- 管理建议

---

## 6. 报告生成模块职责

报告生成模块负责把多个处理模块的结构化输出统一写成月报。

### 6.1 主要职责

- 按标准模板分章节生成
- 对缺失字段给出提示
- 标识待人工确认内容
- 生成 Markdown 初稿

### 6.2 输入

- 项目概况字段
- 财务字段
- 欠费字段
- 合规字段
- 运维字段
- 安保字段
- 公用事业字段
- 治理字段

### 6.3 输出

- 章节 Markdown
- 完整报告 Markdown
- 缺失项清单
- 审核提示清单

---

## 7. 推荐工作流

建议第一阶段采用如下编排流程：

```text
用户选择物业 + 月份
  -> 主控 Agent 读取配置
  -> 扫描基础资料目录
  -> 扫描月度资料目录
  -> 检查资料完整性
  -> 调用财务与欠费处理模块
  -> 调用合同与合规处理模块
  -> 调用运营与现场处理模块
  -> 调用治理与待决事项处理模块
  -> 汇总结构化结果
  -> 调用报告生成模块
  -> 输出报告初稿 + 缺失项 + 审核提示
```

---

## 8. 主控 Agent 的输入与输出

### 输入

- property_name
- report_month
- 目录配置
- 字段字典
- 报告模板

### 输出

- 一份 `report_id`
- 一套结构化字段结果
- 一份报告初稿
- 一份缺失资料提示
- 一份需人工审核提示

---

## 9. 多目录处理策略

对于分散在不同目录的资料，主控 Agent 不应“全部混读”，而应有顺序和优先级。

### 9.1 优先级建议

#### 优先级 1：结构化主数据

优先读取：

- 项目主数据表
- 合同台账
- 执照台账
- 财务 Excel
- 欠费 Excel

原因：

- 准确率高
- 可直接入字段

#### 优先级 2：半结构化资料

再读取：

- 安保报告 PDF
- 会议资料 PDF
- 工程报告 PDF

原因：

- 用于补足章节内容和事件摘要

#### 优先级 3：非结构化资料

最后读取：

- 图片
- 邮件导出
- 聊天记录导出

原因：

- 噪音更高
- 适合作为补充证据和异常发现

---

## 10. 缺失资料处理建议

主控 Agent 在调用各模块之前，应先检查资料是否齐全。

例如：

- 是否存在财务表
- 是否存在欠费表
- 是否存在合同台账
- 是否存在安保月报
- 是否存在水电统计表

如果缺失，应输出：

- 缺失资料清单
- 仍可生成哪些章节
- 哪些章节只能部分生成

这样比直接失败更实用。

---

## 11. 第一阶段是否真的需要多个 Agent

第一阶段推荐答案是：

### 逻辑上：是协同的

因为确实存在多个职责模块。

### 实现上：不一定要很多独立 Agent

第一阶段完全可以实现为：

- 一个主控 Agent
- 多个内部处理模块

而不是：

- 多个完全自治、彼此来回通信的独立 Agent

这会更稳。

---

## 12. 第二阶段的可扩展方向

如果第一阶段跑通并稳定后，第二阶段再考虑把某些模块升级成真正独立 Agent。

例如：

- Finance Agent
- Compliance Agent
- Operations Agent
- Report Writer Agent
- Review Agent

适用场景：

- 多物业并行处理
- 多用户协作
- 更高频率自动运行
- 更强的可观测性与任务编排

---

## 13. 推荐实施建议

综合考虑第一阶段目标、复杂度和成功率，建议采用以下方案：

### 推荐方案

**一个主控 Agent，统一编排；多个分类处理模块，按类别处理；最终统一生成报告。**

### 这样做的优点

- 范围清晰
- 开发复杂度低
- 便于调试
- 业务容易理解
- 后续可平滑扩展为真正的多 Agent 架构

---

## 14. 第一阶段落地建议

第一阶段可按以下顺序实施：

1. 建立基础资料与月度资料双层目录
2. 固化字段字典与模板
3. 实现主控 Agent 的扫描与编排逻辑
4. 实现 4 个分类处理模块
5. 实现统一报告生成模块
6. 输出 Markdown 初稿并人工审核

---

## 15. 结论

对于“资料分散在多个目录、部分资料并非每月变化、分类不同”的场景，第一阶段最合理的实施方式不是复杂多 Agent 自由协作，而是：

**主控 Agent 统一调度，基础资料与月度资料分层读取，分类处理模块按领域处理，最后统一生成报告。**

这是一种既符合实际业务，又能控制第一阶段复杂度的方案。
