# AI物业报告自动生成实施方案

本文档基于今天已完成的工作整理形成，目标是在**尽量不改变现有物业管理工作流**的前提下，利用 AI 自动采集现有资料并生成物业管理报告初稿，帮助管理处减少整理、汇总和写作工作量。

---

## 1. 文档目标

本方案重点解决以下问题：

- 现有物业资料分散在图片、Excel、PDF、聊天记录、电子邮件等不同载体中
- 月报、会议报告和综合管理报告主要依赖人工收集与撰写
- 数据重复整理，效率低，容易遗漏
- 关键事项如欠费、合同到期、执照到期、重大事件难以及时归纳

本方案的目标不是推翻现有工作方式，而是在现有工作流基础上新增一个 AI 汇总层，实现：

1. 自动读取原始资料
2. 自动抽取关键信息
3. 自动归类到标准报告章节
4. 自动生成物业报告初稿
5. 由物业经理审核后输出正式报告

## 3. 总体原则

本项目建议遵循以下原则：

### 3.1 不做大改造

不要求一线员工改变原有工作方式，不强制切换到新系统录入。现有资料仍可继续通过以下方式产生：

- Excel 表格
- PDF 报告
- 手机拍照图片
- 邮件附件
- 聊天记录导出

AI 只负责在后端读取和整理这些内容。

### 3.2 先半自动，再全自动

优先实现：

- 人工收集资料
- AI 自动抽取和汇总
- 人工审核后发布

不建议一开始就做完全无人审核的自动发报。

### 3.3 先结构化，再写报告

不要让 AI 直接“看一堆文件后自由发挥写报告”，而是先抽取成标准字段，再按模板写报告，这样更稳、更可控、更易审计。

### 3.4 保留来源追溯

每一段报告内容都应能追溯到原始资料，例如：

- 来自哪个 Excel 表
- 来自哪封邮件
- 来自哪张图片
- 来自哪个 PDF

这样便于审核和纠错。

## 4. 现有工作流下的 AI 介入点

在不改变现有业务流程的前提下，建议把 AI 放在以下位置：

### 4.1 资料归档之后

当每月资料陆续形成后，统一放入指定目录：

- 财务报表
- 欠费报表
- 合同与执照资料
- 工单记录
- 现场图片
- 邮件导出
- 聊天记录导出

AI 定时扫描这些目录。

### 4.2 经理写报告之前

在物业经理准备月报或委员会报告前，由 AI 先输出一份初稿，经理只需：

- 核对关键数字
- 补充管理判断
- 确认措辞
- 批准发布

### 4.3 会议准备阶段

在委员会会议前，AI 可提前自动生成：

- 待决事项清单
- 财务摘要
- 欠费重点单位
- 即将到期合同与执照
- 维修与投诉重点事项

## 5. 目标工作流

建议建立如下最小可行流程：

### 步骤 1：收集原始资料

仍按现有方式收集资料，不要求一线改变。

资料类型包括：

- Excel 财务表
- Excel 欠费表
- Utility 表
- 合同台账
- 执照文件
- PDF 报告
- 维修照片
- 巡检照片
- 邮件导出
- 聊天记录导出

### 步骤 2：统一归档

将资料按“物业 + 月份”归档到标准目录。

### 步骤 3：AI 自动抽取

AI 对不同资料执行不同处理：

- 图片：OCR + 图像理解
- Excel：表格字段提取与汇总
- 邮件：事件、金额、日期、审批结论提取
- 聊天记录：投诉、事件、待办、决定提取
- PDF：章节、金额、日期、到期信息抽取

### 步骤 4：写入结构化中间表

所有抽取结果统一进入结构化数据层。

### 步骤 5：规则校验

由规则引擎进行分类与风险判断，例如：

- 超过 2 个季度欠费
- 合同 90 天内到期
- 执照 60 天内到期
- 重大财务偏差
- 重大投诉或事故

### 步骤 6：自动生成报告初稿

AI 按标准模板自动生成月度物业报告。

### 步骤 7：人工审核并发布

由物业经理审核后导出正式报告。

## 7. 建议的目录结构

建议先从文件夹规范开始，不需要先做复杂系统。

每个物业建议采用如下目录：

```text
[物业名称]/
  2026-02/
    01_财务
    02_欠费
    03_合同与执照
    04_维修与工单
    05_安保
    06_公用事业
    07_邮件与聊天
    08_图片
    09_输出报告
```

例如：

```text
Goldenhill/
  2026-02/
    01_财务/
    02_欠费/
    03_合同与执照/
    04_维修与工单/
    05_安保/
    06_公用事业/
    07_邮件与聊天/
    08_图片/
    09_输出报告/
```

文件命名建议如下：

- `财务报表_2026-02.xlsx`

- `业主投诉_漏水_#12-05_2026-02-21.eml`

这样 AI 更容易识别资料类型。

## 9. 报告生成逻辑

AI 报告生成建议分两步执行。

### 9.1 第一步：按章节生成草稿

AI 先单独生成每个章节，例如：

- 财务章节
- 欠费章节
- 合同与执照章节
- 运营维护章节
- 安保章节

这样更易审核，也更方便排查错误。

### 9.2 第二步：合并为完整报告

再将各章节拼装成完整月报，按照统一模板输出。

输出格式建议支持：

- Markdown
- Word
- PDF

## 10. AI 在不同资料类型上的具体作用

### 10.1 图片

AI 可执行：

- OCR 提取图片中文字
- 识别图片中的设备、场景和异常
- 自动生成图片说明
- 判断属于维修、清洁、安全或投诉类问题

适用案例：

- 电梯故障照片
- 墙面渗水照片
- 违停照片
- 公共区域破损照片

### 10.2 Excel

AI 可执行：

- 自动识别表头
- 抽取关键数字和日期
- 汇总金额
- 识别异常值
- 转换为报告描述语言

这是最优先实施的资料类型。

### 10.3 邮件

AI 可执行：

- 提取投诉、报价、审批、续约、到期等信息
- 自动分类到财务、合同、维修、客诉等章节
- 提取附件中的关键内容

### 10.4 聊天记录

AI 可执行：

- 提炼事件摘要
- 识别待办事项
- 判断是否已关闭
- 按投诉、事件、维修、审批分类

建议以“导出文本后批处理”为主，不建议一开始做实时聊天机器人。

### 10.5 PDF / Word

AI 可执行：

- 抽取章节和结构
- 提取金额、日期、到期日
- 摘要会议纪要、安保报告、财务说明
- 提炼可进入月报的段落

输入：

- 工单

- 维护摘要

## 12. 推荐的人机分工

建议明确 AI 与人工的职责边界。

### AI 负责

- 读取文件
- 分类资料
- 提取日期、金额、单位、事件
- 汇总和归类
- 按模板写报告初稿
- 标注缺失项和待确认项

### 人工负责

- 确认关键财务数据
- 确认法律/合规相关措辞
- 判断是否升级为律师追讨或委员会事项
- 增加管理判断和建议
- 审批最终对外版本

## 13. 风险与控制措施

### 13.1 OCR 识别错误

风险：

- 扫描件数字和日期识别错误

控制：

- 财务金额必须人工复核
- 高风险字段设置信心阈值

### 13.2 Excel 格式不统一

风险：

- 不同物业、不同月份表头不一致

控制：

- 尽量逐步统一 Excel 模板
- 先从 1 到 2 个物业做标准化试点

### 13.3 聊天记录噪音过大

风险：

- 大量无效消息干扰提取

控制：

- 仅导出与物业运营相关群组
- 用规则过滤关键词

### 13.4 AI 幻觉或误写

风险：

- AI 在缺数据时编造结论

控制：

- 明确要求“缺失就标注未提供”
- 输出必须关联来源

### 13.5 缺少追溯

风险：

- 无法知道报告段落来自哪里

控制：

- 所有抽取记录保留 source_file 和 source_page/source_row

### 阶段 1：模板与字段固化

目标：

### 阶段 3：自动初稿生成

目标：

- 按模板自动生成月报草稿

### 阶段 4：更多资料接入

目标：

- 接入邮件

- 增加定时生成
- 增加仪表盘与提醒

## 15. 推荐的近期执行计划

建议用 4 周完成一个最小可行试点。

### 第 1 周

- 选定 1 个物业作为试点
- 确认月报模板
- 整理最近 3 个月资料样本
- 建立标准目录结构

### 第 2 周

- 定义字段字典
- 确定 Excel、PDF、图片的抽取规则
- 建立第一版抽取脚本/流程

### 第 3 周

- 跑通“抽取 -> 字段表 -> 报告初稿”
- 与人工原报告对比
- 修正字段遗漏和错误分类

### 第 4 周

- 让物业经理试用
- 收集反馈
- 修订模板与规则
- 决定是否推广到第二个物业

## 16. 本项目最适合的启动方式

如果只选择一个最稳妥的启动方式，建议如下：

### 启动方案

1. 先选 1 个物业
2. 只接入 3 类资料：Excel、PDF、图片
3. 先生成结构化字段
4. 再自动生成月报 Markdown 初稿
5. 保留人工审核

### 原因

- 对现有工作流影响最小
- 技术复杂度最低
- 回报最快
- 最容易验证 AI 是否真正节省时间

## 17. 对应到智能物业管理系统的落地方向

今天整理出的系统模块可与本方案逐步对应：

- 月报管理：自动生成报告
- 知识库：存放历史报告、制度、SOP、合同、执照
- 合规日历：跟踪合同和执照到期
- 欠费催缴：跟踪 arrears 与催缴状态
- 预算管理：分析预算与实际差异
- 工单跟踪：汇总维修和维护状态
- 发票处理：后续可接入财务自动化

也就是说，AI 自动生成物业报告可以作为整个智能物业管理系统的第一个落地场景。

---

## 18. 结论

在现有工作流下，最合理的 AI 实施路径不是重建一个全新平台，而是建立一个“资料采集 + AI 抽取 + 结构化汇总 + 模板生成”的轻量中间层。

这条路径的优势是：

- 不需要大规模改变一线习惯
- 能快速形成月报自动化能力
- 可以逐步扩展到合同、合规、欠费、维修等更多模块
- 可作为未来智能物业管理系统的核心基础能力

当前最建议的下一步不是继续讨论概念，而是马上进入试点准备：

1. 选定试点物业
2. 固化目录结构
3. 固化字段清单
4. 用 1 个月资料跑出第一版自动报告

## 附录 A：字段存储与取回设计

本附录用于补充说明：按照《物业报告字段字典》抽取出来的数据，应该如何存储、如何取回、以及为什么不能只保存成一篇 Markdown 报告。

### A.1 设计目标

字段存储设计需要同时满足以下目标：

- 支持自动生成物业报告
- 支持人工审核与修正
- 支持来源追溯
- 支持后续趋势分析和 dashboard
- 支持未来扩展到合同提醒、欠费预警、运维分析等场景

因此，第一阶段不建议只保存“报告成品”，而应同时保存“结构化字段结果”。

### A.2 推荐存储原则

建议采用以下基本原则：

1. 原始文件不入数据库正文，只保存索引信息
2. 字段抽取结果进入结构化存储
3. 明细类数据单独建表
4. 报告初稿单独保存为 Markdown 或 Word 输出文件
5. 每个关键字段保留来源信息和置信度

一句话概括：

**原始文件保存在文件系统，结构化结果保存在数据库，报告成品单独输出。**

### A.3 推荐的三层存储结构

建议按以下三层组织数据。

#### 第 1 层：原始资料层

用于保存文件索引，不改变原始资料本身。

建议保存内容：

- 文件路径
- 文件类型
- 所属物业
- 所属月份
- 文件哈希值
- 上传时间

用途：

- 知道原始文件在哪里
- 支持来源追溯
- 支持重复文件识别

#### 第 2 层：结构化结果层

这是最核心的一层，用于保存 Agent 按字段字典抽取出的结果。

建议保存内容：

- 报告主信息
- 财务汇总
- 欠费汇总
- 合同与执照汇总
- 维修与运营汇总
- 安保汇总
- 公用事业汇总
- 治理与待决事项汇总

用途：

- 生成月报
- 人工审核
- 趋势分析
- 二次复用

#### 第 3 层：明细与追溯层

用于保存列表型、明细型和来源型数据。

建议保存内容：

- 欠费明细
- 合同明细
- 执照明细
- 维修明细
- 安保事件明细
- 客诉明细
- 字段来源明细

用途：

- 支持逐项核查
- 支持字段级追溯
- 支持按单元、合同、事件查询

### A.4 为什么不能只存 Markdown

如果只保存一篇 Markdown 报告，会有以下问题：

- 无法方便地按字段复核
- 无法按月份做趋势对比
- 无法按物业做横向分析
- 无法准确追溯某个数值来自哪里
- 无法把数据复用于 dashboard、提醒和预警模块

因此，Markdown 只适合作为“输出成品”，不适合作为唯一数据载体。

### A.5 推荐的最小数据库模型

第一阶段建议使用“主表 + 字段表 + 明细表 + 来源表”的结构。

#### 1. 报告主表 `reports`

建议字段：

- report_id
- property_name
- mcst_no
- report_month
- report_type
- status
- generated_at
- reviewed_at

作用：

- 一份报告一个主记录
- 所有章节、字段、明细都挂在 `report_id` 下面

#### 2. 字段结果表 `field_extractions`

建议字段：

- id
- report_id
- field_name
- field_value
- value_type
- confidence_score
- review_required
- review_status
- source_file
- source_page
- source_sheet
- source_row
- extracted_text

作用：

- 存储每个字段的抽取结果
- 支持逐字段审核
- 支持来源追溯

这张表是第一阶段最关键的表。

#### 3. 章节结果表 `report_sections`

建议字段：

- id
- report_id
- section_name
- section_json
- section_markdown
- review_status

作用：

- 存储按章节生成的结果
- 支持按章节审核
- 支持快速拼装整份报告

#### 4. 来源文件表 `source_documents`

建议字段：

- id
- report_id
- file_name
- file_path
- source_type
- file_hash
- uploaded_at
- parsed_status

作用：

- 记录本次报告使用了哪些原始资料
- 支持审计和回查

### A.6 推荐的明细表

### A.7 推荐的主键与取回键

建议统一使用：

- `report_id = [property_code]_[YYYY-MM]_[report_type]`

例如：

- `GOLDENHILL_2026-02_MONTHLY`
- `CAMBIO_2026-02_MONTHLY`

这样做的好处：

- 一眼可读
- 查询简单
- 生成、审核、导出时都可以共用同一键

常见取回方式如下：

#### 按报告取回

适用于生成整份报告：

- 用 `report_id` 查询 `reports`
- 用 `report_id` 查询 `field_extractions`
- 用 `report_id` 查询各明细表
- 再按模板拼装报告

#### 按字段取回

适用于审核某个字段：

- 用 `report_id + field_name` 查询字段结果

例如：

- `report_id = GOLDENHILL_2026-02_MONTHLY`
- `field_name = arrears_total_amount`

可直接取到：

- 字段值
- 来源文件
- 来源页码/工作表
- 抽取原文
- 是否已复核

#### 按物业和月份取回

适用于报表和趋势分析：

- 用 `property_name`
- 用 `report_month` 范围

例如查看最近 12 个月：

- 欠费总额趋势
- 管理基金盈余趋势
- 公用事业成本趋势

### A.8 推荐的技术选型

第一阶段建议用轻量方案，不必上复杂架构。

#### 推荐方案：SQLite

优点：

- 易部署
- 本地可运行
- 无需单独数据库服务
- 适合 Agent V1 试点

适用场景：

- 单物业试点
- 内部团队使用
- 快速验证

#### 第二阶段可考虑：PostgreSQL

适用场景：

- 多物业
- 多用户协作
- 未来接前端、API、仪表盘

#### 不推荐作为长期方案：只用 JSON 或只用 Markdown

原因：

- 查询困难
- 趋势分析困难
- 审核困难
- 追溯困难

### A.9 推荐的数据流

建议采用以下数据流：

```text
原始资料目录
  -> Agent 扫描
  -> 内容抽取
  -> 写入结构化数据库
  -> 生成章节结果
  -> 生成 Markdown 报告初稿
  -> 人工审核修订
```

这个流程的优点是：

- 报告生成可重复
- 抽取结果可复核
- 数据可以复用

### A.10 第一阶段最实用的落地组合

如果第一阶段希望低成本、低复杂度上线，最推荐的组合是：

- 文件系统：存原始资料
- SQLite：存结构化字段和明细结果
- Markdown：存 AI 生成报告初稿

这是目前最适合 Agent V1 的落地方式。

### A.11 后续扩展价值

一旦字段存储层建好，后面可以直接扩展出更多能力：

- 合同到期提醒
- 执照到期提醒
- 欠费趋势看板
- 月度运维 dashboard
- 重点事项自动摘要
- 跨物业对比分析

因此，字段存储设计不仅服务于“生成报告”，也是后续智能物业管理系统的数据基础。
