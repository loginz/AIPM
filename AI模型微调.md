## 一、最容易误判的分界线：微调 vs RAG

这是实际项目中最常见的错误决策。简单记住一句话：

**RAG 补知识，微调改行为。**

如果物业AI不知道「某栋楼的设备型号」，这是知识问题 → 用 RAG 把设备档案接进去。如果物业AI知道设备型号但每次输出格式不一致、语气不对、分类逻辑不稳定，这是行为问题 → 才需要微调。把文档塞进微调是最常见的资源浪费，模型不能可靠地「记住」大量事实。

---

## 二、物业平台具体场景的微调优先级

**值得微调的场景（行为稳定性要求高）：**
- 工单分类输出 JSON 结构（字段、类型、值域必须精确）
- 紧急程度判断（「电梯困人」必须走紧急通道，不能误判）
- 拒答边界（不属于物业职责范围的问题要有固定回答方式）

**不值得微调的场景（换 RAG 或 Prompt）：**
- 查询业主合同条款详情 → RAG
- 回答物业收费标准 → RAG + Prompt
- 多语言支持 → Prompt 调整

---

## 三、数据积累的实用策略

在数据还不够的早期阶段，不要等——用「影子模式」运行：让 AI 生成答案但不直接给用户，人工审核后留下高质量的输入-输出对。这样每一次人工处理都同时是标注数据。等积累到 200-500 条后，第一次微调就有了真实分布的训练集，而不是凭空构造的合成数据。

## 不需要自部署大模型，就能做微调

商业API微调（以OpenAI为代表）已经非常成熟——上传数据、调用API、拿到一个专属的微调模型ID，整个过程不需要GPU、不需要ML工程师，几十行代码完成。对物业平台早期来说，这是正确起点。

---

## 三条关键判断线

**什么时候商业API微调够用：** 工单分类、格式输出、回复风格——这些任务用 GPT-4o mini 微调，500条数据，训练成本可能不超过10美元，效果立竿见影。

**什么时候需要考虑迁移自部署：** 出现以下任一情况时开始评估——月推理费用超过2000美元（自部署长期更便宜）、客户有严格数据合规要求（业主身份证、财务流水不能出境）、需要深度定制模型架构。迁移不是第一天的决定，是业务规模到了之后的优化。

**关于 Anthropic Claude 的特殊情况：** Claude 目前还没有开放标准的微调 API，主要通过企业合作渠道定制。如果你的主力模型是 Claude，目前的替代路径是精心设计 System Prompt + Tool Use 组合，在很多场景可以达到接近微调的效果，等 Anthropic 开放后再迁移。

---

## 物业平台实际建议

早期完全不需要碰 GPU。用 OpenAI Fine-tuning API 验证「微调对工单分类有没有帮助」这个命题，成本低、速度快。等验证了价值、数据量积累到几千条、或者客户对数据主权开始提要求的时候，再讨论是否迁移到云托管开源模型（Qwen2.5 在中文场景很强）或完全私有化部署。把精力放在数据质量和业务逻辑上，基础设施问题留到真正需要解决的时候再解决。
